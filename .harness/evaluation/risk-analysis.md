# 风险分析

> 评估代码变更对系统的风险影响。与 review-checklist.md 综合判定对接。
> 关联评估: `security-check.md` / `architecture-score.md` / `hallucination-check.md`
> 关联约束: `constraints/database.md` / `constraints/production.md` / `constraints/payment.md`

---

## 风险维度

### 1. 变更范围风险（权重 20%）

```yaml
评估:
  单文件修改:
    - 工具类/常量: 低风险
    - DTO/VO: 低风险
    - Controller: 中风险（API 兼容性）
    - Service: 中风险（业务逻辑）
    - Entity/Mapper: 高风险（数据层）
    - 配置类: 高风险
    - Gateway Filter: 高风险（影响全局流量）

  前端文件修改:
    - 样式/CSS: 低风险
    - 组件 UI: 低风险
    - API 调用层（src/api/）: 中风险（接口兼容性）
    - 状态管理（stores/）: 中风险（数据流）
    - 路由配置（App.tsx）: 中风险
    - 布局组件（layouts/）: 低风险

  多文件修改:
    - 同模块内 2-3 文件: 中风险
    - 跨模块 2+ 模块: 高风险
    - 修改 pom.xml 依赖: 高风险
    - 修改 Flyway 脚本: 高风险（不可回滚）
    - 修改 application.yml: 高风险
    - 修改 .env* 环境变量: 高风险

  模块风险权重:
    - 支付模块（yuemo-payment）: ×2
    - 订单模块（yuemo-order）: ×1.5
    - 用户模块（yuemo-user）: ×1.5
    - 网关模块（yuemo-gateway）: ×1.5
    - 其他模块: ×1

评分:
  - 无高风险项: 0-30 分
  - 存在中风险项: 30-60 分
  - 存在高风险项: 60-80 分
  - 多个高风险项 + 跨模块: 80-100 分
```

### 2. 数据一致性风险（权重 20%）

```yaml
检查:
  - [ ] 是否修改了事务边界？
  - [ ] 是否涉及缓存与数据库一致性？
  - [ ] 是否涉及 MQ 消息顺序/幂等？
  - [ ] 是否涉及跨表操作？
  - [ ] 是否涉及余额/库存扣减？
  - [ ] 是否涉及 Redis Key 结构/TTL 变更？（购物车主存储在 Redis）
  - [ ] 是否涉及逻辑删除一致性？（唯一键含 deleted 约束）

风险等级:
  - 修改事务传播行为: 高风险
  - 新增/修改缓存策略: 高风险
  - 修改 MQ 消息格式: 高风险
  - 修改金额/库存计算: 高风险
  - 修改订单状态流转: 高风险
  - 修改 Redis Key 命名/结构: 高风险（缓存失效 + 数据不一致）
  - 修改 Redis TTL 策略: 中风险
  - 违反唯一键含 deleted 约束: 高风险（数据插入失败）

评分:
  - 无数据一致性风险: 0-30 分
  - 存在中风险项: 30-60 分
  - 存在高风险项: 60-80 分
  - 多个高风险项: 80-100 分
```

### 3. 兼容性风险（权重 15%）

```yaml
检查:
  API 兼容性:
    - [ ] 是否修改了 API 路径？
    - [ ] 是否修改了请求/响应字段名？
    - [ ] 是否修改了字段类型？
    - [ ] 是否删除了已有字段？

  数据库兼容性:
    - [ ] 是否 DROP COLUMN（数据丢失）？
    - [ ] 是否修改列类型（可能截断）？
    - [ ] 是否添加 NOT NULL 无默认值？

  MQ 消息兼容性:
    - [ ] 是否修改了消息体格式？（消费者可能无法反序列化）
    - [ ] 是否新增/删除了消息字段？
    - [ ] 是否修改了 Topic 名称？

  Redis 兼容性:
    - [ ] 是否修改了 Key 命名格式？（旧 Key 失效）
    - [ ] 是否修改了 Value 序列化格式？

  前端兼容性:
    - [ ] 是否修改了 API 调用路径？
    - [ ] 是否修改了响应数据结构？

风险等级:
  - 破坏性 API 变更: 高风险（需前端同步上线）
  - MQ 消息格式不兼容: 高风险（消费者异常）
  - Redis Key 格式变更: 高风险（缓存雪崩）
  - 删除数据库字段: 高风险
  - 新增必填字段: 中风险
  - 新增可选字段: 低风险

评分:
  - 无兼容性风险: 0-30 分
  - 存在中风险项: 30-60 分
  - 存在高风险项: 60-80 分
  - 多个高风险项: 80-100 分
```

### 4. 安全风险（权重 15%）

```yaml
检查:
  - [ ] 是否引入了 SQL 注入风险？（${} 拼接）
  - [ ] 是否引入了越权访问风险？（缺少归属校验）
  - [ ] 是否暴露了敏感数据？（密码/手机号/Token）
  - [ ] 是否修改了鉴权/认证逻辑？
  - [ ] 是否修改了支付安全相关代码？
  - [ ] 是否引入了 XSS 风险？（未转义用户输入）

风险等级:
  - SQL 注入: 高风险（CRITICAL）
  - 支付安全违规: 高风险（CRITICAL）
  - 越权访问: 高风险
  - 敏感数据泄露: 高风险
  - 鉴权逻辑变更: 高风险
  - XSS 风险: 中风险

评分:
  - 无安全风险: 0-30 分
  - 存在中风险项: 30-60 分
  - 存在高风险项: 60-80 分
  - 存在 CRITICAL 项: 80-100 分
```

### 5. 性能风险（权重 15%）

```yaml
检查:
  - [ ] 是否新增了循环内的数据库查询（N+1）？
  - [ ] 是否新增了全表扫描查询？
  - [ ] 是否新增了未加索引的查询条件？
  - [ ] 是否移除了缓存？
  - [ ] 是否新增了大对象序列化？
  - [ ] 是否新增了深分页查询？（OFFSET 过大）

风险等级:
  - N+1 查询: 高风险
  - 全表扫描: 高风险
  - 移除缓存: 高风险
  - 深分页: 中风险
  - 缺少 LIMIT: 中风险

评分:
  - 无性能风险: 0-30 分
  - 存在中风险项: 30-60 分
  - 存在高风险项: 60-80 分
  - 多个高风险项: 80-100 分
```

### 6. 回滚难度（权重 15%）

```yaml
评估:
  易回滚:
    - 纯代码修改（git revert）
    - 配置修改（git revert + 确认）
    - 前端修改（重新部署镜像）

  难回滚:
    - 数据库迁移（Flyway 不支持回滚，需逆向迁移脚本）
    - MQ 消息格式变更（需兼容消费，已发送不可撤回）
    - 缓存结构变更（需清理策略）
    - Redis Key 格式变更（需清理旧 Key + 预热新 Key）
    - application.yml / .env 变更（需确认回滚值）
    - 跨服务 API 变更（需协调回滚）

风险等级:
  - 涉及 Flyway 脚本: 高风险（不可自动回滚）
  - 涉及 MQ 消息格式: 高风险
  - 涉及 Redis Key 格式: 中风险
  - 涉及配置变更: 中风险
  - 纯代码/前端: 低风险

评分:
  - 易回滚: 0-30 分
  - 部分难回滚: 30-60 分
  - 多项难回滚: 60-80 分
  - 不可回滚项: 80-100 分
```

## 评分算法

```yaml
各维度评分: 0-100 分（0=无风险, 100=极高风险）

总分计算: 加权求和
  总分 = scope × 0.20
       + data_consistency × 0.20
       + compatibility × 0.15
       + security × 0.15
       + performance × 0.15
       + rollback × 0.15

风险等级映射（与 review-checklist.md 对齐）:
  < 30:  LOW       → 正常上线
  30-60: MEDIUM    → 需要额外验证
  60-80: HIGH      → 需要架构评审 + 回滚演练
  > 80:  CRITICAL  → 需要架构评审 + 回滚演练 + 用户确认

与 review-checklist 综合判定对接:
  LOW      → 可 PASS
  MEDIUM   → WARN
  HIGH     → FAIL（需豁免理由）
  CRITICAL → FAIL（BLOCK）
```
