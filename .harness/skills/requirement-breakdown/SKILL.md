# 需求拆解 Skill

将用户的高层需求拆解为结构化、可执行的开发任务清单，确保需求落地不遗漏。

## 触发条件

当出现以下场景时，必须激活此 Skill：

- 用户提出新功能需求（如"加一个秒杀功能"）
- 用户描述业务场景但未给出具体实现方案
- 用户的需求描述模糊、笼统，需要进一步细化
- 用户要求评估某个需求的实现范围

## 拆解流程

### 第一步：需求澄清

在拆解之前，先对需求进行澄清，确认以下维度：

1. **业务目标**：这个需求要解决什么业务问题？核心价值是什么？
2. **用户角色**：涉及哪些用户角色（C端用户、商家、运营、管理员）？
3. **核心场景**：主要使用场景有哪些？正常流程和异常流程分别是什么？
4. **边界条件**：需求中未明确但必须处理的边界情况
5. **非功能性要求**：性能、安全、并发、兼容性等方面的要求

如果用户的需求描述不足以完成拆解，必须使用 AskUserQuestion 工具向用户提问澄清，而不是自行假设。

### 第二步：影响面分析

基于项目架构，分析需求涉及的影响面：

**后端影响面**（Spring Boot 3.2 + MyBatis-Plus）：

| 分析维度 | 检查项 |
|---------|--------|
| 模块影响 | 需要修改 yuemo-modules 下的哪些模块（user/product/order/payment/cart/promotion）？ |
| 数据库变更 | 是否需要新建表、加字段、改索引？参考 `yuemo-backend/sql/` |
| API 变更 | 是否需要新增或修改 API？遵循 `.harness/docs/api-conventions.md` |
| 缓存影响 | 是否涉及 Redis 缓存策略变更？ |
| 消息队列 | 是否需要引入 RocketMQ 消息？ |
| 网关/鉴权 | 是否需要修改 yuemo-gateway 的路由或鉴权规则？ |
| 限流熔断 | 是否需要 Sentinel 规则调整？ |

**前端影响面**（React 19 + Ant Design 5）：

| 分析维度 | 检查项 |
|---------|--------|
| 页面变更 | 需要新增或修改 `yuemo-frontend/src/pages/` 下的哪些页面？ |
| 组件变更 | 是否需要新增公共组件到 `components/`？ |
| 状态管理 | 是否需要新增 Redux store 或修改现有 store？ |
| API 对接 | 前端请求层 `utils/request.ts` 是否需要调整？ |
| 类型定义 | 是否需要新增或修改 `types/` 下的类型？ |

**基础设施影响面**：

| 分析维度 | 检查项 |
|---------|--------|
| 部署变更 | Docker Compose 编排是否需要调整？ |
| CI/CD | `.gitlab-ci.yml` 流水线是否需要修改？ |
| Nginx | 反向代理/负载均衡配置是否需要调整？ |

### 第三步：任务拆解

将需求拆解为具体的开发任务，每个任务必须满足以下要求：

1. **原子性**：一个任务只做一件事，可独立完成和验证
2. **可执行**：描述清晰，开发者无需再猜测实现方式
3. **有依赖**：明确标注任务之间的前置依赖关系
4. **可验证**：每个任务都有明确的验收标准

任务按以下分类组织：

```
📋 需求：{需求标题}

🎯 业务目标
{一句话描述业务目标}

👥 涉及角色
- {角色1}：{该角色在此需求中的操作}

📐 影响面分析
- 后端模块：{列出受影响模块}
- 前端页面：{列出受影响页面}
- 数据库：{列出表变更}
- 基础设施：{列出基础设施变更}

📝 任务清单

【数据库】
  ☐ DB-1: {任务描述}
     - 依赖：无
     - 验收：{验收标准}

【后端】
  ☐ BE-1: {任务描述}
     - 依赖：DB-1
     - 验收：{验收标准}
  ☐ BE-2: {任务描述}
     - 依赖：BE-1
     - 验收：{验收标准}

【前端】
  ☐ FE-1: {任务描述}
     - 依赖：BE-1（API 就绪）
     - 验收：{验收标准}

【联调】
  ☐ INT-1: {任务描述}
     - 依赖：BE-2, FE-1
     - 验收：{验收标准}

【基础设施】
  ☐ INFRA-1: {任务描述}
     - 依赖：无
     - 验收：{验收标准}

⚠️ 风险与注意事项
- {风险1}
- {风险2}

🔗 任务依赖关系图
{用文本描述任务间的依赖链路}
```

### 第四步：优先级排序

对拆解出的任务进行优先级排序：

- **P0 - 必须完成**：核心流程，缺少则功能不可用
- **P1 - 应该完成**：重要但非核心的增强功能
- **P2 - 可以延后**：优化类、体验类的改进

## 约束规则

1. **不遗漏数据库变更**：任何涉及数据持久化的需求，必须先拆解数据库任务
2. **API 先行**：后端任务中，API 定义必须先于业务逻辑实现
3. **前后端解耦**：前端任务依赖后端 API 就绪，但可并行开发（使用 mock）
4. **遵循项目规范**：所有任务必须符合 `.harness/rules/` 下的编码规范
5. **安全合规**：涉及用户数据、支付、权限的需求，必须标注安全审查点
6. **不自行假设**：需求不明确时，提问而非假设

## 输出要求

- 使用中文输出
- 任务编号格式：`{类别缩写}-{序号}`（如 DB-1, BE-1, FE-1）
- 每个任务描述不超过两句话
- 验收标准必须具体可测，避免"功能正常"这类模糊描述
- 依赖关系必须指向具体的任务编号
